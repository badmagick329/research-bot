# Research Bot: Architecture and Operations

## Purpose
This document is the operator/developer reference for how the system runs today.

## Runtime architecture
- Host runtime: Bun CLI + Bun API + Bun workers.
- UI runtime: Vite app in `apps/web`.
- Infra runtime: Postgres + Redis (Docker recommended locally).
- Architecture style: clean architecture boundaries (`core` -> `application` -> `infra`).

## Pipeline model
- Stages: `ingest -> normalize -> embed -> synthesize`.
- Stage payload is run-scoped: `runId`, `taskId`, `symbol`, plus diagnostics context.
- Symbol resolution happens before ingestion and carries canonical identity through the run.
- Ingestion writes durable evidence rows (news, metrics, filings).
- Synthesis reads persisted evidence and writes immutable snapshots.

## Snapshot and thesis behavior
- `snapshot` is read-only and never triggers pipeline work.
- `refresh-thesis` enqueues `synthesize` only.
- `refresh-thesis` reuses persisted run evidence; it does not call news/metrics/filings providers.
- Snapshot thesis is stored as markdown.
- Evidence Map is deterministic and programmatic:
  - label mapping (`N#`, `M#`, `F#`, `R#`) is generated by code in synthesis.
  - LLM does not own Evidence Map content.

## LLM and embeddings
- Chat/synthesis provider is selectable via `LLM_PROVIDER` (`ollama` or `openai`).
- Embeddings are Ollama-based and independent from chat provider.
- Cross-run memory retrieval is embedding-based, symbol-scoped, and excludes current run.
- Memory citations (`R#`) are support context; current-run evidence remains primary.

## Failure and degradation model
- Ingestion requires at least one successful source call.
- Ingestion hard-fails on Alpha Vantage `rate_limited` responses.
- Normalize/embed failures degrade and continue with stage diagnostics.
- Synthesis still materializes snapshot output with quality alerts when possible.
- Diagnostics propagated to snapshot:
  - `metricsDiagnostics`
  - `providerFailures`
  - `stageIssues`
  - `identity`

## Idempotency, retries, and pacing
- HTTP retries: 2 retries (3 attempts total).
- BullMQ retries: 2 retries (3 attempts total), exponential backoff.
- Provider rate limiting is Redis-backed and shared across workers.
- Default provider pacing:
  - Alpha Vantage: 1 request/sec
  - Finnhub: 1 request/sec
  - SEC EDGAR: 1 request/sec
- Queue idempotency key format: `${symbol}-${stage}-${hour}`.
- Use `enqueue --force` to bypass hourly dedupe.

## API and web behavior
- Web API routing precedence:
  - if `VITE_API_BASE_URL` is set, browser calls that base URL directly.
  - if not set, web uses relative `/api` and Vite proxy.
- Snapshot refresh endpoint:
  - `POST /api/snapshots/:symbol/refresh-thesis`
  - optional body: `{ "runId": "..." }`

1. API: `bun run api`
2. Worker: `bun run worker`
3. Web: `bun run web:dev`

## Core commands
- `bun run enqueue --symbol AAPL`
- `bun run enqueue --symbol AAPL --force`
- `bun run refresh-thesis --symbol AAPL`
- `bun run refresh-thesis --symbol AAPL --run-id <RUN_ID>`
- `bun run snapshot --symbol AAPL --prettify`
- `bun run status`

## Operational rules
- Run CLI/API/worker from host terminal; Docker is for infra services.
- Restart worker after provider/resolver/config changes that affect runtime behavior.
- Restart worker after changing limiter env values.
- Restart worker after changing `LLM_PROVIDER`, model env vars, or related API keys.
- Embed model dimension must match vector storage expectation (currently 1024).
- Queue names and custom job ids must not include `:`.

## Troubleshooting
- No snapshot appears:
  - check worker logs for failed jobs.
  - verify API and worker are both running.
  - use `enqueue --force` to bypass stale idempotent jobs.
- Web `/api/*` connection errors:
  - align `API_PORT` and `VITE_API_PROXY_TARGET`.
  - clear/fix `VITE_API_BASE_URL` if set incorrectly.
- Weak/empty cross-run memory:
  - confirm earlier runs completed embed stage.
- Thesis-only rerun:
  - use `refresh-thesis` (not `enqueue`) to avoid provider repull.

## Migrations
- Standard flow:
  1. edit schema: `src/infra/db/schema.ts`
  2. generate SQL: `bunx drizzle-kit generate`
  3. review files in `drizzle/`
  4. apply: `bun run db:migrate`
- Use versioned migrations (`migrate`), not `push` outside throwaway local prototyping.

## Current limitations
- Polling UI only (no SSE/WebSocket).
- No AuthN/AuthZ.
